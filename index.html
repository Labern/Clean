How Claude Code actually works

No magic — just a model with hands. Claude Code is me, running in your terminal (or here, in a browser session like this one), with real access to read files, edit them, run shell commands, and check my own work. Same brain as chat, but instead of describing code I write it.

🧠

It's an agent loop

I read your request, decide which tool to use (read a file, run a command, search the repo), look at the result, and decide the next step — repeating until the task is done, not just one shot.

🛠️

Real tools, real effects

Read/Write/Edit touch your actual filesystem. Bash runs actual commands. Nothing is simulated — which is why risky actions (deletes, force-pushes) get a confirmation step.

🗂️

Context is the fuel

I work from what's in the conversation plus what I read from disk. The more relevant context I have — code, docs, your conventions — the less I guess.

🧩

Skills & subagents

For specialized jobs (code review, security review, deep research) I can invoke purpose-built skills, or spin up focused subagents for parallel/independent digging.

What I'd recommend, concretely

  • Start every real project with a CLAUDE.md — it's the single highest-leverage thing you can do. It's the difference between me guessing your conventions and just knowing them.
  • Let me explore before I edit. For unfamiliar codebases, ask me to read/search first — bad assumptions compound fast in agentic edits.
  • Keep tasks scoped. "Fix this bug" beats "refactor the whole app" — small, verifiable steps mean I catch mistakes before they spread.
  • Use git. Even solo, commits give us a undo button. I'll never force-push or rewrite history without asking.
  • Tell me when I'm wrong. Corrections stick — I keep a memory of feedback so I don't repeat the same miss next session.

Setting up your own CLAUDE.md

Think of it as onboarding docs for a new hire who reads fast but knows nothing about your project's quirks. Run /init in a repo and I'll draft one by reading the codebase — but here's the shape that works well: 

# CLAUDE.md

## Project
One paragraph: what this is, who it's for, the stack.

## Commands
- build: npm run build
- test:  npm test
- lint:  npm run lint

## Conventions
- State management: Zustand, not Redux
- Styling: Tailwind only, no CSS modules
- Tests live next to source as *.test.ts

## Gotchas
- The payments service needs STRIPE_KEY in .env.local
- Don't touch /legacy — slated for deletion, not maintained

## Do NOT
- Don't add new dependencies without asking
- Don't run migrations against prod

Keep it short and honest — a stale CLAUDE.md that lies about your stack is worse than none at all. Update it the moment a convention changes, the same way you'd update a README.
